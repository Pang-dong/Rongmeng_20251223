## 三、架构分层说明

该项目采用经典的 **四层架构**，从上到下依赖单向流动：

```
┌─────────────────────────────────────────┐
│           View 层（XAML + Code-Behind）   │  ← 只做 UI 渲染与事件转发
├─────────────────────────────────────────┤
│           ViewModel 层                   │  ← 持有状态、Command、调用Service
├─────────────────────────────────────────┤
│           Service 层                     │  ← 封装业务逻辑，不依赖UI
├─────────────────────────────────────────┤
│    Infrastructure 层（LH + Interface）   │  ← TCP通信、协议解析、配置文件
└─────────────────────────────────────────┘
```

### 数据流向

```
用户点击按钮
   → View 触发 Command（绑定到 ViewModel）
   → ViewModel 调用 Service 方法
   → Service 调用 ClientApi.Send(IDocommand)
   → TCP 发送字节流到设备
   → 设备响应 → ClientApi 解析 → 发送 CommandResponseMessage（Messenger）
   → DeviceBusinessService 接收并处理业务逻辑
   → WeakReferenceMessenger 广播 Messages（日志字符串）
   → ViewModel 订阅 → 更新 UI 状态
```

## 四、核心设计模式

### 1. MVVM 模式（CommunityToolkit.Mvvm）

ViewModel 继承 `ObservableObject`，属性通过 `SetProperty` 触发双向绑定，命令使用 `[RelayCommand]` 特性或手动构造 `AsyncRelayCommand`。

```csharp
// 属性绑定示例
public bool IsConnected
{
    get => _isConnected;
    set
    {
        if (SetProperty(ref _isConnected, value))
            RefreshAllCommands(); // 联动刷新命令可用状态
    }
}

// 异步命令示例
ConnectCommand = new AsyncRelayCommand(Connect, CanConnect);
```

**要点：** View 的 Code-Behind 只负责无法通过绑定实现的操作（如 `PasswordBox`、视频帧渲染、窗口关闭）。

---

### 2. 消息总线模式（WeakReferenceMessenger）

模块间解耦通信，避免强引用依赖。项目定义了两种消息：

| 消息类 | 用途 |
|--------|------|
| `Messages` | 日志字符串广播（Service → ViewModel） |
| `CommandResponseMessage` | 设备响应命令传递（ClientApi → DeviceBusinessService） |

```csharp
// 发送消息（Service层）
WeakReferenceMessenger.Default.Send(new Messages("连接成功"));

// 订阅消息（ViewModel层）
WeakReferenceMessenger.Default.Register<Messages>(this, (r, m) =>
{
    Application.Current.Dispatcher.Invoke(() => AddLog(m.Value));
});
```

---

### 3. 工厂方法模式（SelectFactory）

隐藏 `FactoryDocommand` 的多个构造函数，提供统一创建入口，调用方只需传入参数类型：

```csharp
// 调用方不需要知道具体构造细节
IDocommand cmd = SelectFactory.CreateDocomandStringAnd(MessageTypes.Command, CommandType.SetLicenceNo, code);
IDocommand cmd = SelectFactory.CreateDocomandInt(MessageTypes.Command, type, intVal);
```

---

### 4. 接口隔离 + 模板方法（IDocommand）

协议命令抽象为接口，`Docomand` 为基类（持有字段），`FactoryDocommand` 为实现类（负责序列化/反序列化逻辑）：

```
IDocommand（接口，定义ToBuffer/FromBuffer契约）
    └── Docomand（基类，持有协议字段）
            └── FactoryDocommand（实现类，多构造函数 + 协议编解码）
```

---

### 5. 单例模式（FFmpegDecoder）

使用 `Lazy<T>` 实现线程安全的懒加载单例，保证全局只有一个解码器实例：

```csharp
private static readonly Lazy<FFmpegDecoder> _lazyInstance =
    new Lazy<FFmpegDecoder>(() => new FFmpegDecoder());

public static FFmpegDecoder Instance => _lazyInstance.Value;
```

---

### 6. 策略模式（StationConfig + 动态按钮）

工站配置存放于外部 JSON 文件（`Chinese/StationConfig.json`），运行时按工站名称筛选，动态生成按钮列表，每个按钮绑定独立的 `AsyncRelayCommand`：

```csharp
// 配置驱动的按钮生成
foreach (var item in items)
{
    CameraControls.Add(new CameraControlItem
    {
        Content = item.Title,
        ConfigData = item,
        Command = new AsyncRelayCommand(
            async () => await RunGenericTest(item),
            () => IsConnected && !IsAutoTesting
        )
    });
}
```

新增测试项只需修改 JSON，无需改代码。

---

### 7. 观察者模式（TcpDeviceinfo / CameraControlItem）

数据模型继承 `ObservableObject`，支持直接绑定到 XAML 控件，属性改变自动刷新 UI：

```csharp
public class TcpDeviceinfo : ObservableObject
{
    public string IPAddress { get => _ipAddress; set => SetProperty(ref _ipAddress, value); }
    public int Port { get => _port; set => SetProperty(ref _port, value); }
}
```

---

### 8. 静态工具类模式（ConfigManager / InvokeMESInterface）

无状态的工具功能封装为 `static class`，直接调用不需要实例化：

```csharp
// 配置读写
ConfigManager.Save(config);
var config = ConfigManager.Load();

// MES接口调用
string result = await InvokeMESInterface.PostToMesAsync(url, "GetUserLoginInfo", args);
```

## 六、通信协议设计（三层封包）

自定义二进制协议，结构如下：

```
┌────────────────────────────────────────────┐
│ Layer 1: 帧头 (16字节)                      │
│  MagicNum(2) | Version(1) | DeviceID(1)    │
│  PacketCount(1) | PackID(1) | Reserved(4)  │
│  Status(1) | BCC校验(1) | DataLength(4)    │
├────────────────────────────────────────────┤
│ Layer 2: 消息包 (MsgPacket)                 │
│  PacketType(4) | MsgLength(4) | CmdPacket  │
├────────────────────────────────────────────┤
│ Layer 3: 命令包 (CmdPacket)                 │
│  CommandType(2) | Payload(N字节)            │
└────────────────────────────────────────────┘
```

- **Payload** 可以是空、int32、JSON字符串或原始字节
- **BCC校验** 对 MsgPacket 逐字节异或

## 七、视频流处理流程

```
TCP接收线程（Receive）
    → 数据写入 MemoryStream 缓冲区
    → ParseLoop 解析完整帧
    → 帧入 Queue<IDocommand>（最多5帧，超出自动丢弃）

帧处理线程（StartReadBuffer）
    → 从 Queue 取帧
    → 调用 FFmpegDecoder.Instance.DecodeFrameToBitmap()（H264/H265解码）
    → 回调 ILHviedoApiCallBack.GetBitmapImg(Bitmap)

MainWindow（实现 ILHviedoApiCallBack）
    → Dispatcher.Invoke 切到UI线程
    → 写入 WriteableBitmap → 渲染到 Image 控件
```

## 八、自动化测试流程设计

主窗口支持"自动测试模式"，通过 `TaskCompletionSource<bool>` 实现异步等待用户判定：

```csharp
// 暂停等待用户点击 PASS/FAIL
_userInputSignal = new TaskCompletionSource<bool>();
bool passed = await _userInputSignal.Task;

// 用户点击后触发
private void OnUserJudgmentReceived(string result)
{
    _userInputSignal?.TrySetResult(result == "PASS");
}
```

## 九、以此框架搭建新项目的步骤

1. **复制基础层（LH目录）**：`ClientApi`、`ConfigManager`、`MessageType`、`TcpDeviceinfo`、`ReturnResult` 原样复用。

2. **定义业务协议枚举**：在 `MessageType.cs` 中添加新的 `CommandType` 枚举值。

3. **创建 Interface/Model**：定义业务数据模型（如 `AppConfig`、`StationTestItem` 等）。

4. **实现 Service 层**：继承 `DeviceBusinessService` 思路，在 `ProcessBusinessLogic` 中处理响应 switch-case。

5. **配置 JSON 驱动界面**：在 `StationConfig.json` 中填写工站名称、按钮标题、十六进制命令。

6. **编写 ViewModel**：继承 `ObservableObject`，手动声明属性，使用 `AsyncRelayCommand`/`RelayCommand`。

7. **XAML 纯绑定**：所有状态通过 `{Binding}` 连接 ViewModel，Code-Behind 只处理视频渲染等特殊场景。

8. **日志与消息**：统一通过 `WeakReferenceMessenger.Default.Send(new Messages(msg))` 广播，ViewModel 订阅后显示到日志列表。

## 十、框架优缺点评估

**优点：**
- 代码量小，结构清晰，适合 1-2 人小团队快速开发
- 配置驱动界面，新增测试项不需要改代码
- MVVM + Messenger 完全解耦 UI 与业务
- TCP通信、视频解码、MES接口均已封装完毕可直接复用

**注意事项：**
- `PasswordBox` 无法直接绑定，需要通过 `Action` 回调处理
- 视频帧渲染必须在 UI 线程执行，注意 `Dispatcher.Invoke` 的使用
- `WeakReferenceMessenger` 使用弱引用，订阅者需保持存活，避免被GC回收
- 大型项目建议引入 DI 容器（如 Microsoft.Extensions.DependencyInjection）替代手动 new
